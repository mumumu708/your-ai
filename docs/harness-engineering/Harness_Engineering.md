# Harness Engineering 深度解读：理论、案例与实践指南

## 一、引言：为什么你需要了解 Harness Engineering

2026 年初，一个全新的工程概念正在快速凝聚——**Harness Engineering**。它不是某个单独工具的用法说明，而是一整套围绕"如何让 AI 编码 Agent 可靠、高效地在真实生产环境中工作"的系统性方法论。

本文档基于两篇核心文献进行深度解读：

- **OpenAI 官方博客**：*Harness engineering: leveraging Codex in an agent-first world*（作者 Ryan Lopopolo，OpenAI Technical Staff，2026-02-11）——这是 OpenAI 内部一支团队用**零行手写代码**在五个月内构建了一个百万行级内部产品的实战经验总结。
- **Charlie Guo 的综合分析文章**：*新兴的 Harness Engineering 指南*（翻译版），汇集了 OpenAI、Stripe、OpenClaw（Peter Steinberger）、Anthropic、Cloudflare 等多家公司和个人的实践，映射了当前 AI 编码 Agent 领域正在汇聚的共同模式。

这两篇文章一篇是**深度单案例实战报告**（OpenAI 团队从零开始的全过程），另一篇是**多案例跨组织的模式总结**。二者互为补充，共同勾勒了 Harness Engineering 的全貌。

## 二、核心概念：什么是 Harness Engineering

### 2.1 定义

**Harness**（直译"马具"或"线束"）在这里指的是：约束、工具、文档和反馈循环的集合体，它们共同保持 AI Agent 高效运行并保持在正确轨道上。

**Harness Engineering** 这一术语由 Mitchell Hashimoto（Terraform、Ghostty 的创建者）提出，其核心理念是：

> "每次你发现 Agent 犯了一个错误，你就花时间设计一个解决方案，使 Agent 再也不会犯同样的错。"

这不是一个被动的过程，而是一套主动的、持续迭代的工程实践。

### 2.2 与传统软件工程的本质区别

| 维度 | 传统软件工程 | Harness Engineering |
|------|-------------|-------------------|
| **核心活动** | 人编写代码 | 人设计环境，Agent 编写代码 |
| **工程师角色** | 制造者（Maker） | 环境构建者 + AI 管理者 |
| **代码质量保障** | Code Review、手动测试 | 架构护栏 + 自动化 Linter + Agent 自验证 |
| **知识管理** | 文档（经常腐烂） | 仓库内版本化知识体系（活的文档） |
| **扩展方式** | 加人 | 加 Agent 并行度 + 优化 Harness |
| **瓶颈** | 编码速度 | 人类注意力和环境成熟度 |

### 2.3 为什么叫"Harness"而不是别的

这个类比非常精确。Harness（马具/挽具）的功能是：

1. **约束方向**——马可以自由奔跑，但马具确保它沿着正确的道路拉车
2. **传递力量**——马的力量通过马具传递到车辆上，产生有用功
3. **保护安全**——防止马脱缰失控

对应到 AI Agent：

1. **架构约束**——Agent 可以生成任意代码，但 Harness 确保它只在允许的架构边界内工作
2. **工具赋能**——Agent 的能力通过 Harness 提供的工具（CI、Linter、MCP 服务器）被放大
3. **防护机制**——防止 Agent 产生不可维护的"slop"代码

## 三、理论框架：工程师角色的分裂

### 3.1 从"制造者"到"双角色"

Charlie Guo 的文章提出了一个关键观察：工程师的工作正在**分裂成两个截然不同的部分**。

**第一部分：构建环境（Building the Harness）**

这是基础设施层面的工作。当 Agent 遇到障碍时，工程师不是去帮它写代码，而是反思："环境中缺少了什么，以至于 Agent 无法自主完成这个任务？"

OpenAI 团队的切身体会是：

> "早期进展比我们预期的要慢，不是因为 Codex 不够能力，而是因为环境规格不足。Agent 缺乏有效运行所需的工具、抽象和内部结构。"

**第二部分：管理工作（Managing the Work）**

这是日常运营层面的工作——规划任务、分配给 Agent、审查输出、做架构决策。这类似于一个技术经理的工作，但管理对象从人变成了 Agent。

### 3.2 两个部分的关系

这两个部分不是先后顺序的，而是**同时进行、互相驱动的**：

- Agent 的失败暴露了环境的缺陷 → 驱动 Harness 改进
- 更好的 Harness 降低了管理摩擦 → 允许更高并行度
- 更高并行度产生更多失败数据 → 进一步驱动 Harness 改进

这形成了一个**正向飞轮**。

## 四、Harness 的四大支柱（理论层）

综合两篇文章的内容，Harness Engineering 的核心实践可以归纳为四大支柱：

### 4.1 支柱一：架构作为护栏（Architecture as Guardrails）

**核心原则**：通过严格的架构约束限制 Agent 的解决方案空间，而不是放任其自由发挥。

这一原则是反直觉的。通常我们认为给 AI 更多自由度会得到更好的结果。但在大规模代码生成中，情况恰恰相反：

> "增加对 AI 生成代码的信任和可靠性，需要**约束**解决方案空间，而不是扩展它。"——Birgitta Böckeler（Martin Fowler 网站）

**理论基础**：

- Agent 在大型解决方案空间中容易"局部最优化"——选择当前看起来合理但长期不可维护的方案
- 明确的架构边界将"创意空间"限制在安全范围内，同时在该范围内给予完全自由
- 这类似于编程语言的类型系统：约束越强，能排除的错误类别越多

### 4.2 支柱二：工具作为基础与反馈（Tools as Foundation and Feedback）

**核心原则**：Agent 需要与人类工程师相同的工具访问权，而且工具不仅扩展 Agent 的能力——它们提高 Agent 已有能力的可靠性。

**理论基础**：

- 工具是 Agent 的"感知器官"——没有工具，Agent 就是在"盲飞"
- 工具反馈构成了 Agent 的学习信号——Linter 错误消息既是约束也是教学
- 工具链的质量直接决定了 Agent 自主性的上限

### 4.3 支柱三：文档作为记录系统（Documentation as System of Record）

**核心原则**：文档不是静态参考资料，而是活的、版本化的、机械验证的反馈循环。

**理论基础**：

- Agent 每个会话都是"无状态"的——它不记得上次犯过什么错
- 文档（尤其是 AGENTS.md）是跨会话的"持久记忆"
- 每次 Agent 犯错就更新文档 = 将"经验"编码为"制度"

### 4.4 支柱四：编排而非仅仅委托（Orchestration, Not Just Delegation）

**核心原则**：高效使用 Agent 不是简单地"把任务丢给它"，而是需要规划、并行化和持续质量控制。

**理论基础**：

- 规划的质量直接决定了 Agent 执行的质量——"垃圾进，垃圾出"
- 并行化有两种模式（有人值守 vs 无人值守），选择取决于 Harness 的成熟度
- "对 slop 说不"是一个必须主动维护的纪律，否则系统会自然退化

## 五、深度案例分析

### 5.1 案例一：OpenAI 内部团队——零手写代码的百万行产品

**背景**：

- 2025 年 8 月，从空的 Git 仓库开始
- 起初 3 名工程师，后增长到 7 名
- 核心约束：**零行手写代码**（人类引导，Agent 执行）

**过程详解**：

**阶段 1：初始脚手架**

第一次提交是由 Codex CLI（使用 GPT-5）生成的：仓库结构、CI 配置、格式化规则、包管理器设置、应用框架。工程师提供的不是代码，而是一组已有的设计文档和约束描述。

**阶段 2：发现瓶颈不在 Agent 能力**

早期进展很慢。团队很快意识到：Agent 不是不会写代码，而是环境没有为它准备好条件。缺乏工具、缺乏抽象、缺乏内部结构。

**阶段 3：深度优先构建 Harness**

团队转向"深度优先"策略：把大目标拆成小构建块（设计、代码、审查、测试），先让 Agent 构建这些块，再用这些块去搭建更复杂的功能。

**阶段 4：让应用对 Agent "可读"**

这是 OpenAI 文章中最核心的工程创新之一。团队做了以下关键投入：

1. **应用可按 Git worktree 启动**——每个 Codex 会话可以启动并驱动一个独立的应用实例。这意味着 Agent 不仅能写代码，还能运行应用、点击 UI、观察行为。

2. **Chrome DevTools Protocol 接入 Agent 运行时**——创建了专门的 skill，让 Codex 可以截图、选择 DOM 元素、触发 UI 路径、检查前后状态差异。Agent 从此可以做视觉验证，而不仅仅是运行单元测试。

3. **完整可观测性栈**——日志、指标、追踪都通过一个本地可观测性栈暴露给 Codex。每个 worktree 都有自己的隔离栈。Agent 可以看到完整的运行时行为。

**阶段 5：知识体系建设**

团队试过"一个巨大的 AGENTS.md"方案，失败了。原因很有教育意义：

- **上下文是稀缺资源**：巨大的指令文件挤占了任务、代码和相关文档的上下文空间
- **什么都重要 = 什么都不重要**：当所有东西都标记为"重要"时，Agent 只能做局部模式匹配
- **瞬间腐烂**：没有机械验证的文档必然过时
- **难以验证**：一个单体文件无法做覆盖率、新鲜度、所有权检查

最终方案：**AGENTS.md 是目录，不是百科全书**。大约 100 行，指向 `docs/` 目录中更深层的真实来源——设计文档、架构地图、执行计划、质量评分等。所有这些都在仓库中版本化。

**阶段 6：架构执行**

每个业务域被分为固定的层集（Types → Config → Repo → Service → Runtime → UI），依赖方向严格验证，跨域引用被禁止。这些约束通过自定义 Linter 和结构化测试机械执行——而这些 Linter 本身也是 Codex 生成的。

**阶段 7：自主性提升**

到了后期，给定一个 prompt，Agent 可以完成从"验证当前状态"到"复现 Bug → 录屏 → 修复 → 验证修复 → 录屏 → 开 PR → 响应反馈 → 检测构建失败 → 在需要判断时升级给人类 → 合并"的完整闭环。

**阶段 8：熵与垃圾回收**

Agent 会复制仓库中已有的模式——包括不够好的模式。久而久之，会产生漂移。最初团队每周五花 20% 的时间手动清理"AI slop"，很快发现无法扩展。最终方案：将"黄金原则"编码到仓库中，构建循环清理流程——由 Agent 自动执行的"垃圾回收"。

**关键数据**：

- 5 个月，百万行代码
- 团队从 3 人增长到 7 人时，**人均吞吐量反而增加**
- 单次 Codex 运行可持续工作超过 6 小时（通常在工程师睡觉时）
- 每个工程师平均每天 3.5 个 PR

### 5.2 案例二：Stripe Minions——万人企业的 Agent 规模化

**背景**：Stripe 是拥有 10,000+ 员工的支付基础设施公司，其内部编码 Agent 叫做 Minions。

**运作模式**：

1. 开发者在 Slack 中发布任务
2. Minion Agent 在隔离的、预热的"devbox"中运行——与人类工程师使用的完全相同的开发环境，但与生产环境和互联网沙盒化
3. Agent 通过名为 **Toolshed** 的集中 MCP（Model Context Protocol）集成连接到 **400+ 个内部工具**
4. Agent 编写代码、通过 CI、打开一个准备好供人类审查的 PR
5. 开发者仅在审查阶段重新进入循环

**每周输出**：超过 1,000 个合并的 PR。

**关键设计决策**：

- **Agent 与人类使用完全相同的环境**——不是事后整合，而是从一开始就共享相同的工具、上下文和访问权限
- **MCP 作为统一接口**——400+ 工具通过一个标准化的协议暴露给 Agent
- **无人值守并行化**——这是目前最成熟的"无监督"Agent 使用模式，但前提是极高的 Harness 投入

### 5.3 案例三：Peter Steinberger / OpenClaw——独行者模式

**背景**：Peter Steinberger 是 OpenClaw 的创建者，一个人在一个月内产生了 6,600+ 次提交，同时运行 5-10 个 Agent。

**关键行为模式**：

1. **发布自己没有逐行阅读的代码**——听起来很疯狂，但他作为"仁慈独裁者"守住了架构
2. **在启动执行前与 Agent 花大量时间规划**——规划是最关键的投入
3. **在 Discord 只谈架构和大决策，不谈代码**——代码是 Agent 的事
4. **作为"大师木匠"存在**——他知道好的东西长什么样，Agent 负责锯和胶

Pragmatic Engineer 的 Gergely Orosz 的评价："他给我的印象是一个软件架构师，将项目的高级结构保持在头脑中。"

### 5.4 案例四：Anthropic 长运行 Agent 的 Harness 实践

**背景**：Anthropic 工程团队发表了关于长运行 Agent 有效 Harness 的研究。

**核心发现**：

1. **每个新 Agent 会话都没有记忆**——这是 Agent 的根本限制
2. **解决方案是结构化进度文件**——类似于从未见面的工程师之间的"轮班交接笔记"
3. **用 JSON 而不是 Markdown 跟踪特征状态**——因为 Agent 更不容易不恰当地编辑结构化数据
4. **"初始化 Agent"生成全面的功能列表**——一个 Web 应用可能有 200+ 个独立功能，每个都有明确测试步骤，全部初始标记为"失败"

这种预先分解防止了两个常见陷阱：Agent 试图一次完成整个项目，或过早宣布胜利。

## 六、实践指南：如何在你的团队落地 Harness Engineering

### 6.1 第一步：审计你的环境

在做任何事情之前，先回答这些问题：

| 检查项 | 问题 | 目标状态 |
|--------|------|----------|
| 架构约束 | 你的代码库有明确的模块边界和依赖方向吗？ | 有，且通过 Linter/CI 机械执行 |
| 工具可访问性 | 你团队的内部工具有 CLI 或 MCP 接口吗？ | Agent 可以直接调用所有关键工具 |
| 测试基础 | 你的测试套件运行速度和覆盖率如何？ | 快速（分钟级）、覆盖核心路径 |
| 文档状态 | 你有 AGENTS.md 吗？有架构文档吗？ | 有，且定期更新、版本化 |
| CI/CD 成熟度 | 你的 CI 管道能在分钟内给出反馈吗？ | 能，且反馈信息对 Agent 友好 |

### 6.2 第二步：建立 AGENTS.md

从小处开始。一个基本的 AGENTS.md 包含：

```markdown
# AGENTS.md

## 项目概览
[一两句话描述项目是什么]

## 构建命令
- 安装依赖：`pnpm install`
- 开发运行：`pnpm dev`
- 运行测试：`pnpm test`
- Lint 检查：`pnpm lint`

## 架构概览
[指向 ARCHITECTURE.md]

## 编码约定
- 使用 TypeScript strict mode
- 数据在边界处用 Zod 解析验证
- 依赖方向：Types → Config → Repo → Service → Runtime → UI
- 禁止跨域直接引用

## 常见陷阱
- [Agent 第一次犯错时添加]
- [Agent 第二次犯错时添加]
- ...

## 深度文档
- 设计文档：docs/design-docs/
- 执行计划：docs/exec-plans/
- 产品规格：docs/product-specs/
```

**关键纪律**：每次 Agent 犯错，就更新这个文件。这不是可选的，这是 Harness Engineering 的核心飞轮。

### 6.3 第三步：建立架构护栏

**具体做法**：

1. **定义分层架构**——不需要从零设计，参考 OpenAI 的模式（Types → Config → Repo → Service → Runtime → UI）并根据你的项目调整

2. **编写自定义 Linter 规则**——关键点：Linter 的错误消息应该同时作为**修复指令**。不要只说"违反了架构约束"，要说"这个模块不应依赖 Service 层，请通过 Types 层的接口来传递数据"

3. **编写结构化测试**——验证依赖方向、验证模块边界、验证命名约定

4. **让 Agent 来生成这些 Linter**——OpenAI 团队的 Linter 本身就是 Codex 生成的，这形成了良性循环

### 6.4 第四步：让工具对 Agent 可访问

**优先级排序**：

1. **第一优先级**：版本控制（git）、构建工具、测试运行器、Linter
2. **第二优先级**：部署工具、监控/可观测性工具、数据库迁移工具
3. **第三优先级**：设计系统参考、外部 API 文档、产品规格

**实施方式**：

- CLI 工具是最直接的接入方式
- MCP 服务器提供更丰富的结构化接口
- 确保工具的输出对 Agent 友好（结构化、有明确的错误信息和修复建议）

### 6.5 第五步：建立规划先行的工作流

**具体做法**：

1. **每个任务先写 Plan，再写代码**——这是 Cloudflare 的 Boris Tane 强调的最重要实践：

   > "这种规划和执行的分离是我做的唯一最重要的事。它防止了浪费努力，让我控制架构决策，并以最少的 token 使用产生显著更好的结果。"

2. **Plan 的格式**：
   - 目标：要实现什么
   - 约束：什么是不允许的
   - 步骤分解：具体的实施步骤
   - 验收标准：怎么算完成
   - 已知风险：可能遇到的问题

3. **Plan 也交给 Agent 生成**——但人类必须审查和批准后才启动执行

### 6.6 第六步：建立质量门控

**"对 slop 说不"的具体实践**：

1. **保持与人类编写代码相同的审查标准**——Agent 输出更快不意味着可以降低标准
2. **关注架构级问题而非语法级问题**——语法交给 Linter，人类关注"这个抽象在正确的层次吗？""这个模式六个月后会不会成为维护噩梦？"
3. **建立"黄金原则"并自动执行**——将品味编码为规则
4. **定期运行"垃圾回收"**——检测和清理 Agent 引入的模式漂移

### 6.7 第七步：逐步提升并行度

**从低到高的并行化路径**：

| 阶段 | 并行度 | 前提条件 |
|------|--------|----------|
| 起步 | 1 个 Agent，有人值守 | 基本的 AGENTS.md + CI |
| 初级 | 2-3 个 Agent，有人值守 | 架构护栏 + Linter |
| 中级 | 3-5 个 Agent，半有人值守 | 完整工具链 + 结构化文档 |
| 高级 | 5-10 个 Agent，部分无人值守 | 成熟 Harness + 高信任度 CI |
| 专家 | 无人值守 Agent 舰队 | Stripe 级别的基础设施投入 |

## 七、开放问题与当前局限

两篇文章都诚实地指出了当前 Harness Engineering 面临的未解决问题：

### 7.1 长期代码健康

如何防止"功能正确但难以维护的代码"在代码库中累积？OpenAI 团队已经开始用"垃圾回收 Agent"来应对，但承认这仍是新兴实践。Agent 生成的代码与人类编写的代码在累积方式上有本质不同——它会复制仓库中已有的模式，包括不好的模式。

### 7.2 大规模验证

Agent 会在没有适当端到端测试的情况下将功能标记为完成。即使有浏览器自动化，视觉和工具访问的限制意味着某些 Bug 会被漏掉。Anthropic 的研究也发现了同样的问题——如果没有明确的 prompt 提示，Agent 认识不到某些东西不工作。

### 7.3 改造（Retrofit）问题

所有成功案例都涉及**新建项目**或从头开始构建 Harness 的团队。将这些技术应用到十年旧的代码库——没有架构约束、不一致的测试、零散的文档——是一个更复杂的问题。Böckeler 将其比作"在从未有过静态分析工具的代码库上运行它——你会淹没在警报中"。

### 7.4 文化采用

两篇文章共同强调的一点是：这些东西不会偶然发生。**必须有人去做这些工作**——创建 Harness、定义流程、基于有效反馈持续更新。这需要组织层面的投入和文化转变。

---

## 八、关键认知转变

阅读完这两篇文章后，以下几个认知转变是最重要的：

### 8.1 瓶颈不在 Agent 能力，在环境设计

这是最核心的洞察。当 Agent "做不好"的时候，第一反应不应该是"模型还不够好"，而应该是"我的环境缺少了什么"。

### 8.2 约束是乘数，不是阻碍

在人类优先的工作流中，严格的架构规则可能感觉"迂腐"。但对于 Agent，"一旦编码，它们立即适用于所有地方"。约束不是限制了 Agent 的能力，而是放大了它的可靠性。

### 8.3 文档是代码

在 Harness Engineering 中，文档不是附属品，而是系统的一部分。它是 Agent 的"上下文入口"。如果文档烂了，Agent 的表现就会烂。

### 8.4 投资复合

每次 AGENTS.md 更新都防止一类未来的失败。每个自定义 Linter 教导每个未来的 Agent 会话。你通过 MCP 暴露的每个工具使每个后续任务更快。前期成本显著，但回报加速。

### 8.5 人类价值在更高抽象层

正如大师木匠的比喻：你的价值不在于锯和胶（写代码），而在于知道好的东西长什么样（架构判断和品味）。Steinberger 不谈代码，只谈架构和大决策。这才是 AI 时代工程师的核心竞争力。

---

## 九、总结：一门正在实时形成的新学科

Harness Engineering 不是一个工具，不是一个框架，而是一套**围绕"如何让 AI Agent 在真实工程环境中可靠工作"的系统性方法论**。它借鉴了软件架构中的经典挑战（分层、解耦、约束），融合了团队管理中的经典智慧（规划、委托、质量控制），并加入了 AI 时代特有的新维度（上下文工程、Agent 可读性、渐进式自主性）。

OpenAI 的案例证明了这套方法论在极端条件下的可行性——零行手写代码构建百万行产品。Stripe、OpenClaw、Anthropic、Cloudflare 的案例证明了它在不同规模、不同场景下的普适性。

这些实践仍在快速演变，但汇聚的方向已经足够清晰，值得每一个工程团队认真对待和投入。正如 Charlie Guo 的总结：

> "这是一门真正实时形成的新学科。剧本仍在编写中，但它的形状正在变得清晰。"
