# 也许是对 OpenClaw 记忆最深度的解析

## 〇、写在前面

OpenClaw（原名Moltbot，原原名Clawdbot，原因显而易见）近期爆火。

OpenClaw能够爆火，抛开客观环境和营销drama不谈，我们认为技术层面上主要有3个值得一提的亮点：
- 高度自治，7*24自主运行，危险而强大（long horizon agent，LHA）
- 全平台接入，易用性高（Channel层）
- **持久化记忆与上下文管理**（**本篇主要讨论**）

记忆发展迅猛，从25年中旬到现在一年不到，该领域发展迅速，正在成为 Agent 基础设施中变化最快的层之一。

其中 **MemOS** 在 26 年 2 月率先尝试对 OpenClaw 适配的记忆系统，当然后续也有其他尝试扩展的方式，但是都大同小异，因此本文将分析 OpenClaw 原生 Memory 与 MemOS 扩展后的 Memory 来看下该类 Memory 系统的发展。

---

## 一、OpenClaw Memory

OpenClaw 原生 Memory 是一个文件优先、完全本地的记忆系统，记忆写入依赖 LLM 主动性，组织结构扁平透明。

[原文档包含架构图，建议查看原飞书文档]

### 1.1 设计理念

OpenClaw 采用 **文件优先 (file-first)** 设计：所有持久化记忆都保存在本地 Markdown 文件中。

这种设计的核心优势是透明：用户和 **Agent** 都可以直接查看、编辑记忆文件，配合 Git 做版本管理，也更容易定位和追踪问题。

### 1.2 工作区与记忆文件

OpenClaw 的工作区（默认 `~/.openclaw/workspace/`）主要包含两类内容：

#### **1.2.1 身份与配置文件**（通过 bootstrap 引导生成，见 1.3）：

- `IDENTITY.md` — Agent 的名字、类型、风格、专属 emoji；初始为空，在第一个session由用户与Agent一起填
- `USER.md` — 用户/所有者的背景信息
- `SOUL.md` — Agent 的行为哲学与灵魂内核，包括边界意识、自我定调等；系统鼓励Agent自我迭代这个文件
- `AGENTS.md` — 代理最核心的操作手册，规定了每次会话必做的事，记忆管理策略，各种规范等。
- `TOOLS.md` — 一个环境特定的本地备忘录，记录机器相关信息，不定义工具怎么用（这是Skills的定义）
- `HEARTBEAT.md` — 用户和Agent可编辑的待检清单，供Agent在心跳轮次时读取决定行为，为空则跳过心跳API调用

> 👍 **重要创新**：这组配置文件是 OpenClaw 重要的创新，**结合拟人化与功能性**，确实对 Agents 实际端到端体验有一个体感上的不同。
>
> **衍生思考**：来自谷歌的研究显示，推理模型的能力提升可能来自其内部模拟的**"思想社会"（Society of Thought）** — 一种由多个具有不同**个性**和**专业**背景的内在视角组成的虚拟多智能体社会，通过辩论、质疑与协作来解决复杂问题。
>
> 大概的意思是，人类的个性在解决问题上确实发挥着作用。性格和专业需要分成两个维度来建模。未来可能通过不同性格和专业组成的 Claw Team 是一个可以研究的方向。（有点类似之前炒作的 moltbook 论坛）
>
> 来源：https://arxiv.org/pdf/2601.10825

**Bootstrap 引导与 SOUL.md**

首次运行时，OpenClaw 会创建 `BOOTSTRAP.md`，通过引导对话帮助用户完成初始配置：先确定 Agent 身份（名字、风格等），再记录用户信息，最后共同完善 `SOUL.md`。引导完成后 `BOOTSTRAP.md` 会删除。

`SOUL.md` 的价值在于它是纯文本、可直接编辑：用户可以随时修改偏好，Agent 下次启动即可读取并生效。

#### **1.2.2 记忆文件**（`MEMORY.md` 与 `memory/*.md`）

其中 `memory/*.md` 会在 Memory Flush 时按日期写入（`memory/YYYY-MM-DD.md`），已存在文件会追加而非覆盖，形成可追溯的时间线记忆。

值得注意的是即使 MEMORY.md 并非始终载入。也走的是按需语义检索。系统提示里有一段行为指令告诉代理："在回答任何涉及过往工作、决策、日期、人物、偏好或待办的问题之前，先对 MEMORY.md 和 memory/*.md 跑 memory_search，再用 memory_get 拉取需要的行。"

### 1.4 记忆写入与召回

**写入有两条路径**，本质都是对以上文件的更新：
- LLM 在对话中主动决定是否记录
- 接近压缩阈值时触发 Memory Flush，提醒保存关键信息

同时记忆也会建立 embedding 索引，存在本地 SQLite 里。**在文件变更时**对记忆文件进行 chunk & embedding。默认每个 chunk 400 token，相邻 chunk 之间有 80 token 的重叠。

支持四种 embedding 来源。默认是 `auto` 模式，按优先级依次尝试：本地模型（node-llama-cpp，自动下载 GGUF 格式的 embeddinggemma-300M）→ OpenAI（text-embedding-3-small，1536 维）→ Gemini（gemini-embedding-001）→ Voyage（voyage-4-large）→ 全部不可用则禁用。

**召回通过两个工具完成**：
- `memory_get(path, from, lines)` 读取特定文件的指定行范围。
- `memory_search(query)` 做搜索返回 top 片段（带路径、行号、相似度分数）

搜索使用 hybrid search。向量搜索（cosine similarity）占 70% 权重，BM25 关键词搜索（基于 SQLite FTS5 全文索引）占 30% 权重，两个分数加权融合排序。候选集倍率是 4 倍，比如你要 top 6 结果，它会先从两种方法各取 24 个候选再融合。最低相似度阈值默认 0.35，低于这个分数的结果会被过滤掉。这些参数都可以在配置里调（但估计一般人不会配，都是用默认值）。

### 1.5 上下文管理

OpenClaw 提供了基本的长上下文管理策略，包括旧内容剪枝、历史摘要压缩与压缩前刷写记忆。

但是目前基本只用来做最终上下文超限前的**兜底**，尽量保留任务连续性与关键决策信息。

### 1.6 特点总结

总体来看，OpenClaw Memory 的记忆特点是，以普通文本文件保存，代理每天写日记，人能直接打开看、能手动改、能用 Git 追溯变化。

整套系统完全跑在本地，不上传到任何云端服务，记忆属于用户自己。

## 二、MemOS 扩展

MemOS 是比较传统的记忆 SaaS 服务，强调结构化写入与智能检索，因此在对 OpenClaw 的记忆扩展上也是复用了 MemOS 的基建。

### 2.1 设计理念与关键机制

**MemOS 本身是类 RAG 记忆服务**，在机制上，MemOS 主要有三点：结构化写入，做语义提炼后写入；多路检索，结合向量召回、意图解析和迭代扩展，提高命中率；异步调度，先快速写入，再后台重组，平衡响应速度与记忆质量。

同时，MemOS 将偏好信息作为独立记忆类型处理（显式/隐式），具备一定的演化追踪能力。

### 2.2 OpenClaw 插件集成方式

OpenClaw 提供了插件系统（支持不同 `kind` 和生命周期钩子），**这使 MemOS 这类外部记忆系统可以接入。**

MemOS 当前通过 **lifecycle 插件**接入：
- `before_agent_start`：检索记忆并通过 `prependContext` **强制**注入上下文
- `agent_end`：将本轮对话异步回传存储

需要注意的是，该插件 `kind` 为 `lifecycle` 而非 `memory`，因此不会技术性替换 OpenClaw 原生 `memory_search` / `memory_get`，两套记忆机制在运行中是共存关系。

### 2.3 实验效果

根据 MemOS 自身的测评报告，集成 MemOS 插件后，在 LOCOMO Benchmark 上，在多个维度上获得了显著提升。

LOCOMO Benchmark 主要评判跨会话、多轮次（平均 300 轮）的对话历史记忆能力，数据偏向生活场景。但是对于 OpenClaw 这类 LHA 系统或 Coding Agent 等生产力场景，更高轮次、更庞大上下文、更复杂的任务依赖，**LOCOMO 的参考性可能有限**。

| 评测指标 | 提升效果 |
|----------|----------|
| **Token 消耗** | **降低 72%+** |
| **模型调用次数** | **降低 59.5%** |
| **长期对话准确率** | **从 0.2373 提升到 0.3168** |
| **跨会话任务轮次** | **减少约 53%** |

## 三、对比总结

| 维度 | OpenClaw 原生 | MemOS |
|------|---------------|-------|
| **设计理念** | 文件优先（file-first） | 记忆操作系统（memory OS） |
| **存储** | 本地 Markdown + SQLite | Neo4j + 向量数据库 |
| **存储结构** | 扁平文档分块 | 层次化图/树（自动聚类 + 摘要） |
| **写入触发** | LLM **主动**记录 + 压缩前自动触发 | 每次对话结束后**自动**捕获 |
| **写入质量** | 直接存储原文/分块 | LLM 提取结构化记忆 |
| **检索方式** | 向量 + BM25 混合，按需调用 | 五路并行 + LLM 意图解析，每次自动注入 |
| **运行方式** | **完全本地，零依赖** | **云端服务，需 Neo4j/Redis 等** |
| **透明度** | Markdown 可读可编辑可 Git | 云端图数据库，用户黑盒 |
| **部署** | **数据不离开本机** | **对话发送到 MemOS Cloud** |

## 四、一些对未来的思考

### 4.1 Workflow 记忆 vs. Agentic 记忆：可控性与自主性的取舍

OpenClaw Memory vs. MemOS 两者的设计哲学有点类似 Agentic vs Workflow。

MemOS 在记忆的结构化抽象、智能组织和偏好管理上更强大。它的写入触发、总结逻辑、更新规则都体现了「类 RAG」系统的特征，行为高度可控，Token 使用高效。

OpenClaw Memory 的复杂度远不及 MemOS，但它的设计是刻意为之：让 LLM 自行决策记忆的生成与获取，用最直接透明的 Markdown 文件组织信息，贴合人类开发者已有的工作范式。

### 4.2 MemOS 的扩展真的有用吗？

短期看有用，目前 MemOS 的扩展本质上，还是对模型能力**打补丁**的逻辑：
- 补充模型 Agentic 能力不足，自主召回准召不足的问题。
- 无遗忘曲线的问题：由于其记忆文件会随着时间天级叠加，记忆文件会越积越多。长时间使用会导致记忆的效果下降。MemOS 有一些原有的基建做这部分能力的补充。

长期来看，**当前 MemOS** 的实现是远不足够的。这是一个随着模型能力提升，边际效益越来越低的实现。

### 4.3 那么 File Base 的记忆系统会是未来的趋势吗？

是 & 也不是。我们回顾人类记忆软件（也就是各种文档软件）的发展历史，最开始，基于本地的 Markdown，Word，到后来，发展到云文档的 Google Doc，飞书文档。

那么未来，适合 Agent 和人类一起协作的软件会不会存在呢？**我认为是会的。基于几个原因：**

- **可靠性：**OpenClaw Memory 存在一个结构性风险：由于记忆数据完全存储在本地，一旦 OpenClaw 因异常操作导致自我删除，记忆数据也会被一并抹除，（就是所谓的我的龙虾养死了）目前 OpenClaw 的回滚机制仍在研发中，云端 Memory 系统（如 MemOS）可以作为这一问题的潜在解决方案。

- **共享：**当前文件系统存储在本地，无法与其他 Agent 和人类共享。当然，目前单 Agent 可能改自己的文件就已经足够，但是未来走向大规模 Agent 协作，它们需要共享上下文和决策记录。更进一步，人类也需要随时审阅、修正 Agent 的记忆，而不是在本地文件系统里翻找 Markdown。这就要求记忆层从"单机文件"升级为一个多方可读写的协作空间。

- **权限：**在商业场景中，权限管理是刚需。一旦 Agent 进入企业级部署，记忆层必然需要一套类似云文档的权限体系：角色、分组、审计日志，缺一不可。

- **多模态：**当前记忆几乎只覆盖文本。但 Agent 的工作场景越来越多模态，截图、白板草图、语音对话、代码执行结果的可视化。如果这些都丢了，只留下文字摘要，信息损失会很大。下一代记忆需要原生支持图片、音频片段、结构化数据的存储和检索。

> 👍 在这里给自己打一个广告，对构建下一代 Agent 记忆系统感兴趣的有志之士，欢迎联系我们合作、共建。

### 4.4 对 OpenClaw 爆火的复盘

OpenClaw 的成功有两个值得关注的因素：

**降低 AI 使用门槛**。回顾近几年的发展：2022 年末 ChatGPT 需要用户精心编写 prompt；2023 年 RAG 和 Cursor 等工具拉近了与 LLM 的距离；2024 年至今 Agent 系统蓬勃发展，AI 的使用不再局限于专业人士。OpenClaw 通过全平台接入（WhatsApp、Telegram、Web）进一步降低了门槛。其启示在于：*好的产品未必一开始就高瞻远瞩，紧跟技术发展、持续向着便捷与易用迭代同样可以成功。*

**构建了生态**。OpenClaw 不仅是一个 Agent，更像是一个操作系统或 App Store。通过 Skill 定义，用户可以让 OpenClaw 在遇到研发需求时先调用飞书 MCP 读取文档、再启动 Claude Code 写代码。这种在生态层面的重定义值得我们思考：未来半年与 AI 应用层相关的工作，如何布局与取舍。
