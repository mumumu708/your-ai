# AI Agent 记忆系统方案对比（2026）

> 基于 OpenClaw 源码分析和行业调研，对比当前主流 AI Agent 记忆系统方案，涵盖 Mem0、Zep/Graphiti、Letta、A-Mem 等。

> 💡 **结论**：OpenClaw 的记忆系统属于 2024-2025 年的"扎实工程方案"，在文件持久化和混合检索方面做得很好，但在知识图谱、时序建模、记忆自进化等方面已落后于新一代方案。

## 二、第一梯队：专用记忆基础设施

### 2.1 Mem0 — 最成熟的通用记忆层

[Mem0](https://mem0.ai/) 是目前最知名的 AI Agent 记忆平台，有论文（[arXiv 2504.19413](https://arxiv.org/abs/2504.19413)）支撑，提供开源自部署和托管云服务两种模式。

**核心架构：三存储融合**

[原文档包含架构图，建议查看原飞书文档]

**关键设计：**
- **Graph Memory（Mem0g）**：2026 年 1 月发布的增强版，用知识图谱捕获实体间的关系结构，能理解"用户之前说喜欢 Python，现在转向 Rust"这种偏好变更
- **分层记忆**：user / session / agent 三级隔离，比 OpenClaw 的 agent-only 粒度更细
- **主动记忆管理**：不是简单存储，而是自动提取、合并、冲突消解、过期清理——是"策展"

**性能数据：**

| 指标 | 对比基线 | 提升幅度 |
|------|----------|----------|
| LLM-as-Judge 准确率 | vs OpenAI 原生 | +26% |
| P95 延迟 | vs OpenAI 原生 | -91% |
| Token 成本 | vs OpenAI 原生 | -90%+ |
| Graph Memory 增益 | vs Mem0 base | +2% overall |

> 🎁 **vs OpenClaw**：Mem0 的记忆是"结构化提取"后的语义单元，OpenClaw 是"原始 Markdown 分块"。Mem0 能处理偏好变更和实体关系，OpenClaw 不能。

### 2.2 Zep / Graphiti — 时序知识图谱

[Zep](https://www.getzep.com/) 的核心组件 [Graphiti](https://github.com/getzep/graphiti) 是一个时序感知的知识图谱引擎，有论文（[arXiv 2501.13956](https://arxiv.org/abs/2501.13956)）。

**核心架构：三层子图**

[原文档包含架构图，建议查看原飞书文档]

**关键设计：**
- **双时序模型（Bi-temporal）**：每条边记录"事件发生时间"和"录入时间"，支持 point-in-time 查询（如"上周三我们讨论了什么"）
- **实时增量更新**：新数据即时融入图谱，不需要批量重算
- **高级重排**：RRF + MMR + 图距离重排 + Cross-encoder 四级重排序

**性能数据：**

| 指标 | 对比基线 | 提升幅度 |
|------|----------|----------|
| LongMemEval 准确率 | vs baseline | +18.5% |
| 响应延迟 | vs baseline | -90% |
| LoCoMo J-score | vs Mem0 (75.71) | 76.60 (+0.89pp) |

> 🎁 **vs OpenClaw**：OpenClaw 的时间感知仅靠文件名日期 + 指数衰减函数，非常粗糙。Zep 的双时序模型能精确回答时序问题，OpenClaw 不能。

### 2.3 Letta（原 MemGPT）— Memory-First Agent

[Letta](https://github.com/letta-ai/letta) 将记忆提升为 Agent 的一等公民，核心理念是"LLM as Operating System"。

**演进路线：**

| 时间 | 阶段 | 核心能力 |
|------|------|----------|
| 2023 | MemGPT 论文 | 自编辑记忆（core / archival / recall） |
| 2024 | Letta Framework | Memory Blocks + 状态化 Agent 运行时 |
| 2025 | Letta Code | Memory-First 编程 Agent |
| 2026.01 | Conversations API | 跨并行体验的共享记忆 |
| 2026.02 | **Context Repositories** | **Git-based 记忆版本管理** |

**Context Repositories — 最新方案（2026.02）：**

这是 Letta 最新发布的重大更新，[用 git 做记忆版本管理](https://www.letta.com/blog/context-repositories)：
- **MemFS**：记忆存储在 git-backed 的 markdown 文件系统中
- **自动版本化**：每次记忆修改自动 git commit，附带描述性 commit message
- **子 Agent 并行写入**：每个子 Agent 用 git worktree 隔离，merge 解决冲突
- **程序化上下文管理**：Agent 可以写脚本重组自己的记忆（"learning in token-space"）

[原文档包含架构图，建议查看原飞书文档]

**Benchmark 表现：** 在 LoCoMo benchmark 上，Letta 用简单 Filesystem Agent 达到 **74.0%**（GPT-4o mini），超过 Mem0 的 68.5%（graph variant）。

> 🎁 **vs OpenClaw**：两者的 Markdown-First 思路相似，但 Letta 走得更远——加入了 git 版本控制、子 Agent 并行写入、程序化上下文管理。OpenClaw 的记忆只能追加写入，不能回溯版本。

## 三、第二梯队：学术前沿

### 3.1 A-Mem — 自进化记忆网络

[A-Mem](https://arxiv.org/abs/2502.12110) 受 Zettelkasten 笔记法启发，发表于 NeurIPS 2025（[GitHub](https://github.com/agiresearch/A-mem)）。

**核心创新：记忆是活的**

[原文档包含架构图，建议查看原飞书文档]

- **原子化笔记**：每条记忆是结构化 Note（关键词 + 标签 + 上下文描述 + 链接）
- **自进化**：新记忆加入时，LLM 自动更新旧记忆的上下文描述和链接关系
- **动态链接**：基于 embedding 相似度 + LLM 推理建立记忆间连接
- **无预定义操作**：不像 OpenClaw 那样有固定的"flush / search / get"操作流

> 💡 **vs OpenClaw**：A-Mem 的记忆会随新信息自我演化（更新旧认知、重建链接）。OpenClaw 的记忆写入后是死数据，只能被检索，不会自我更新。

### 3.2 Agentic Memory — 统一长短期记忆

[论文 arXiv 2601.01885](https://arxiv.org/abs/2601.01885)，将记忆管理直接建模为 Agent 的策略学习问题。

- 长期记忆（LTM）和短期记忆（STM）统一管理框架
- 记忆操作暴露为 tool-based actions，Agent 通过 RL 学习何时存 / 取 / 忘
- 自适应优先级排序和遗忘策略——解决"记忆越来越多，检索越来越慢"的问题

## 四、维度对比总结

| 维度 | OpenClaw | Mem0 | Zep / Graphiti | Letta | A-Mem |
|------|----------|------|----------------|-------|-------|
| 存储模型 | Markdown 文件 + SQLite | Vector + Graph + KV | 时序知识图谱 | Git-backed MemFS | Zettelkasten Notes |
| 检索方式 | Hybrid (BM25 + Vector) | 语义 + 图遍历 | 图遍历 + 多级重排 | 文件系统检索 | Embedding + LLM 推理 |
| 时间建模 | 文件名日期 + 指数衰减 | 隐式时间戳 | **双时序模型** | Git 历史 | 时间标签 |
| 记忆进化 | 静态（写入后不变） | 自动合并 / 冲突消解 | 增量图更新 | Agent 自编辑 | **自进化网络** |
| 隔离粒度 | Agent + DM/群组 | User / Session / Agent | Session + Entity | Agent + Memory Block | 无显式隔离 |
| 知识图谱 | 无 | 有（Mem0g） | **核心能力** | 无 | 隐式链接图 |
| 上下文压缩 | **Pre-Compaction Flush** | Token 优化 | 图摘要 | 程序化重组 | 无 |
| 安全防护 | Prompt Injection 检测 | 基础过滤 | 未明确 | 未明确 | 无 |
| 生产就绪度 | 高（平台内集成） | **最高**（托管服务） | 高（开源 + 企业版） | 高（开源框架） | 低（研究阶段） |

## 五、OpenClaw 优势与差距

### OpenClaw 做得好的

1. **Pre-Compaction Memory Flush**：在上下文截断前静默保存，设计巧妙，Letta 后来也借鉴了类似思路
2. **Markdown-First 可调试性**：记忆可读、可编辑、可 git 管理，Letta 的 Context Repositories 也走了这条路
3. **Graceful Degradation**：无 embedding 退回 FTS5，无 sqlite-vec 退回 JS 余弦——在实际部署中非常有价值
4. **Prompt Injection 防护**：LanceDB 插件的注入检测是其他方案普遍缺失的
5. **Hybrid Search**：BM25 + Vector + MMR + Temporal Decay 的组合管线设计成熟

### OpenClaw 的主要差距

1. **无知识图谱**：不能表达实体关系（"Alice 是 Bob 的经理"），Mem0 和 Zep 都有
2. **无时序建模**：指数衰减太粗糙，不能回答"上周三讨论了什么"
3. **记忆不会进化**：写入后是死数据，不会随新信息更新旧认知
4. **无记忆版本控制**：Letta 已用 git 管理记忆历史，OpenClaw 只能追加写入
5. **分层粒度不足**：缺少 user 级记忆（跨 Agent 共享的用户偏好）

## 六、升级建议

如果要升级 OpenClaw 的记忆系统，按优先级排列：

| 优先级 | 方向 | 参考方案 | 预期收益 |
|--------|------|----------|----------|
| P0 | 引入轻量级知识图谱 | Graphiti (Neo4j) / Mem0g | 支持实体关系和时序查询 |
| P1 | 对 memory/ 加 git 版本管理 | Letta Context Repositories | 记忆可回溯、并行写入、冲突解决 |
| P1 | Auto-Capture 加入记忆合并/消解 | Mem0 Consolidation | 偏好变更自动更新旧记忆 |
| P2 | 增加 user 级记忆层 | Mem0 三级隔离 | 跨 Agent 共享用户偏好 |
| P2 | 双时序模型替代指数衰减 | Zep Bi-temporal | 精确时序查询能力 |
| P3 | 探索记忆自进化 | A-Mem Zettelkasten | 长期记忆质量持续提升 |
